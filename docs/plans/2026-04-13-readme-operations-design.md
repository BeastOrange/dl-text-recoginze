# README Operations Design

日期：2026-04-13

## 目标

将根目录 `README.md` 从“项目简介 + 零散示例”升级为可直接执行的操作手册，覆盖：

- 环境创建
- 数据集下载与目录布局
- 中文主线数据处理
- 检测 / 识别 / 端到端训练
- 评估与报告生成
- 结果产物存放位置
- 英文 benchmark 支线

## 约束

- 文档必须严格贴合当前 CLI 和配置文件，不能虚构不存在的命令行为
- 需要明确区分“真实推理/评估”和“手工录入指标归档”
- 需要同时兼容标准 `data/raw/...` 布局与当前仓库内的本地 benchmark 布局
- 不新增新的代码能力，只收口现有使用方式

## 方案比较

### 方案 A：最短快速开始

只保留安装、少量命令和数据集链接。

- 优点：短
- 缺点：仍然不能回答“训练怎么启动、结果在哪”的核心问题

### 方案 B：单 README 完整手册

在根目录 `README.md` 中按“环境 -> 数据 -> 训练 -> 评估 -> 结果目录”重写操作说明。

- 优点：入口单一，最适合课程项目交付和答辩展示
- 缺点：README 会比现在长

### 方案 C：README + 多篇 docs 拆分

README 只放索引，细节拆到 `docs/`。

- 优点：结构清晰
- 缺点：当前仓库用户第一眼拿不到完整操作路径，且维护成本更高

## 采用方案

采用方案 B，并保留英文 benchmark 作为支线章节。

## 设计细节

### 文档结构

README 调整为以下结构：

1. 项目定位与目录结构
2. 环境创建与依赖安装
3. 数据布局、配置文件与下载链接
4. 中文主线：
   - 数据校验
   - detection 数据准备
   - recognition 数据准备
   - 可选 crop 流程
   - 检测训练
   - 识别训练
   - 端到端训练 / 推理
5. 评估与报告：
   - detection 指标归档
   - recognition 指标归档
   - benchmark 评测
   - 端到端图像 / manifest 评估
   - 项目汇总报告
6. 结果目录说明
7. 英文 benchmark 支线
8. 变更记录要求

### 关键澄清

- `data build-rec-lmdb`：保留旧命名，但实际生成 `JSONL manifest`
- `evaluate detector` / `evaluate recognizer`：当前是把指标写入报告，不是自动推理
- `evaluate end2end`：支持真实单图推理、manifest 批量评估和扫参

### 验证

- 检查 README 中引用的路径与当前配置一致
- 重新运行脚本帮助命令与至少一条本地 benchmark 数据命令
- 提交并推送到 `origin/main`
